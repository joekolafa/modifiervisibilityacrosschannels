Modifier Visibility across channels

Four rendering surfaces. Three independent pipelines. Each one wrote its own rules against a visibility model that was never designed for 19 order sources — or for the Drive Thru channel that could stress-test them all.

Section 01

The rendering layer hasn't kept pace with the platform

Toast's visibility model was built for a multi-channel future. The rendering layer wasn't. Each surface — KDS, kitchen print, receipt, POS check view — was built independently and resolves modifier defaults independently, with no concept of where an order came from.

Early days — POS only
One surface, one set of defaults — no concept of channel-specific behavior.
2015 — Online Ordering
The first channel split: some modifier groups needed to stay off guest-facing menus, and the visibility model was built to support it.
2018 — Delivery partner integrations
Grubhub and others introduced a new need: modifier groups authoritative for delivery orders but invisible on POS — creating the dual-axis configuration at the root of this issue.
Today — 19+ order sources and a fault line about to widen

The Drive Thru team is expanding channel visibility to the modifier group and item level. Kiosk visibility is being separated from POS as a distinct channel. Both land directly on the rendering gap this document addresses.

Drive Thru may be the sharpest example of why it matters — guests pull up to the same ordering station, orders are shown on the same KDS, but orderSource shifts depending on whether a staff member or an AI voice integration takes the order.

Section 02

Menu Config & Channel Support

Sequence & swap pricing, size matching, portions, and more are configurable in Toast's menu architecture, but don't behave consistently across channels. A pricing structure that works correctly on POS may charge incorrectly on a guest-facing ordering platform, produce confusing running totals, or not render at all. See module limitations →

Maintaining parallel menus is a tax, not a feature

Some brands skip split-channel entirely and maintain separate menu databases per channel. It solves the rendering problem, but every price change and menu update now has to happen across all of them.

Integration mapping and the cost of structural duplication

When channel visibility can only be controlled at the menu level, the workaround isn't just duplicate modifier groups — it can mean duplicate items, menu groups, modifiers, and categories for the same logical product. That scope matters because enterprise integrations — loyalty platforms, third-party accounting systems, labor and inventory connectors — map to stable item and modifier identifiers across locations. Duplicate structures mean duplicate mappings, and not all integration partners support that cleanly. Changes that should be simple propagate across parallel structures, each with its own integration footprint, and failures don't always surface where the menu decision was made.

⚠
Configure once, render everywhere is the primary value proposition competing enterprise platforms are using against us at this tier. This isn't a nice-to-have — it's closing a gap that has a competitive name on it.🔗

Brands configure modifier group visibility in Toastweb through four checkboxes — POS, Kiosk / Mobile Order & Pay, Toast Online Ordering, and Partners. The UI presents them as equivalent controls. They aren't. Each maps to a different legacy field: POS controls a Visibility enum (ALL, POS_ONLY, NONE), Kiosk maps to a showOnKiosk boolean, Online Ordering to an OrderableOnline enum, and Partners to a GrubhubOrderable enum (YES, NO, INHERITED) — named after the only delivery partner that mattered at the time, quietly repurposed to mean "any third-party partner."

That field name is a record of how the rendering layer evolved — which is to say, it largely didn't. Each new channel got accommodated without the underlying model being revisited. Brands configuring these checkboxes today are working with a UI that abstracts away a distinction the rendering layer still hasn't resolved: there is no unified concept of "channel" that all four surfaces read from. There's a visibility enum, a partner flag, and an orderSource value at runtime — and each surface was written to read whichever of those it happened to look at first.

Those settings control ordering. On the rendering side — what the kitchen, receipt, and check actually display — a different set of rules applies: default modifier behavior, display mode, guest receipt, single selection rules — each a variable the rendering layer has to respect. Proposed fixes account for these.

Drive Thru

Most channels have a fixed order source — a DoorDash order is always API, a walk-in is always IN_STORE. Drive Thru doesn't have that luxury. The scenario at the speaker is always the same. What determines the order source is what happens next — a cashier entering it on POS, or an AI voice integration submitting it through the API. Same station, different pipeline, different visibility rules.

Section 03

The ingredients are all there

Every API order already carries its orderSource. The menu model already distinguishes POS_ONLY from partner-only groups. The data needed to render correctly was always there — it just wasn't read at the right boundaries. Each surface was built independently, each made its own assumptions, and a fix on one never carried to another.

KDS Display
orderSource was never passed into the default modifier resolution layer. Every API order fell back to POS behavior — POS_ONLY groups were treated as authoritative, producing spurious "No X" lines for defaults the partner never touched.

How the order flows

API order → KitchenSelectionFactory.kt (kitchen/framework) → DefaultModifiersHelper.getRemovedDefaults(ancestors, selection, orderSource) → getDefaultModifiersWithSource(item, group, stack, orderSource) → MenuItemHelper.getConsolidatedOptionGroups(item, parentGroup, orderSource)

Root cause

KitchenSelectionFactory built ticket selections without extracting orderSource from the order. The two-arg getRemovedDefaults(ancestors, selection) had no place for it. Every order — POS, API, partner — resolved defaults identically, including POS_ONLY modifier groups that API/partner orders have no visibility into.

What changed — all within toastmobile, merged May 2025

How the order flows

Three entry points in toastmobile, one shared pipeline:

SelectionLineFactoryFactory.kt (toast-common)          ─┐
CustomerReceiptFactoryFactory.kt (orders-view)        ─┤
ReceiptLineBuilderKtUtil.kt (toast-api, 3 builders)  ─┘
  │  all pass filterPosOnlyGroupsForApiOrders = true/false
  ▼
DefaultModifiersWrapperFactory (toast-receipts)
  → SelectionService.getDefaultModifiers(selection, orderSource)
    → MenuService.getDefaultModifiers(item, group, excludePosOnlyGroups)
      → ModifierGroupFinder(excludePosOnlyGroups=true).getModifierGroups()
        → isVisible(): visibility != POS_ONLY (instead of != NONE)

Root cause

ModifierGroupFinder.isVisible() had one rule: visibility != NONE. Correct for POS orders. For API/partner orders, POS_ONLY groups should be excluded because the partner has no visibility into them. The predicate couldn't distinguish order sources — SharedMenuOptionGroupModel only exposes getVisibility(), not the richer isPosOnly() / grubhubOrderable fields available on the concrete MenuOptionGroup in toast-api.
